# 中英文双语文档管理规范

## 1. 文档结构

### 1.1 目录结构
- 中英文文档采用完全相同的目录结构
- 文件名保持一致，仅通过目录区分语言
- 示例：
  ```
  docs/
  ├── en/                     # 英文文档
  │   ├── project-overview/   # 项目概述
  │   └── architecture/       # 架构设计
  └── zh/                     # 中文文档
      ├── project-overview/   # 项目概述
      └── architecture/       # 架构设计
  ```

### 1.2 文件命名规范
- 使用小写字母和连字符（-）分隔单词
- 避免使用特殊字符和空格
- 示例：`system-architecture.md` 而非 `System Architecture.md`

## 2. 文档格式

### 2.1 格式要求
- 使用 Markdown 格式编写所有文档
- 遵循 [GitHub Flavored Markdown](https://github.github.com/gfm/) 规范
- 使用相对路径引用图片和其他资源

### 2.2 模板使用
- 所有文档必须使用统一的模板
- 模板包含：
  - 文档标题
  - 文档版本
  - 最后更新日期
  - 作者信息
  - 内容摘要
  - 正文内容

### 2.3 图片和资源管理
- 所有图片和资源文件放在 `docs/shared/` 目录下
- 使用相对路径引用：`![图片描述](../shared/image.png)`
- 图片命名规范：
  - 使用小写字母和连字符（-）分隔单词
  - 包含清晰的描述性名称
  - 示例：`system-architecture-diagram.png`

## 3. 翻译流程

### 3.1 初始翻译
1. 优先编写英文文档，作为原始版本
2. 然后翻译为中文文档
3. 确保术语翻译的一致性

### 3.2 更新流程
1. 当需要更新文档时，先更新英文原始版本
2. 在英文文档中标记需要翻译的内容
3. 同步更新中文版本
4. 在文档头部的更新日志中记录更新内容和日期

### 3.3 翻译质量要求
- 准确性：确保翻译内容准确无误
- 一致性：术语翻译保持一致
- 流畅性：语言表达自然流畅
- 专业性：使用专业术语

## 4. 术语管理

### 4.1 术语表维护
- 建立统一的术语表，记录所有专业术语的中英文对照
- 术语表定期更新，确保所有团队成员使用一致的术语
- 术语表位置：`docs/shared/glossary.md`

### 4.2 术语使用规范
- 所有文档必须使用术语表中定义的术语
- 首次使用术语时，应在括号中注明英文或中文
- 示例："智能合约（Smart Contract）是区块链上的自动执行合约。"

## 5. 版本控制

### 5.1 文档版本号
- 文档版本号格式：`主版本号.次版本号.修订号`
- 主版本号：重大变更，如文档结构调整
- 次版本号：新增内容或较大更新
- 修订号：小的修正和改进

### 5.2 版本控制流程
- 文档与代码保持相同的版本号
- 每个版本的文档都应归档保存
- 使用 Git 进行文档版本控制
- 在文档头部添加版本信息和更新日志

## 6. 文档审查

### 6.1 审查流程
1. 文档编写完成后，提交给文档负责人审查
2. 审查内容包括：
   - 内容准确性
   - 格式规范性
   - 术语一致性
   - 翻译质量（针对中文文档）
3. 审查通过后，文档正式发布

### 6.2 审查标准
- 准确性：内容准确无误
- 完整性：覆盖所有必要内容
- 清晰度：表达清晰，易于理解
- 一致性：格式和术语保持一致
- 时效性：及时更新，反映最新状态

## 7. 文档工具

### 7.1 编写工具
- 推荐使用 Visual Studio Code 编写 Markdown 文档
- 推荐插件：
  - Markdown All in One
  - Prettier - Code formatter
  - markdownlint

### 7.2 翻译工具
- 推荐使用专业的翻译工具，如：
  - Crowdin
  - Transifex
  - DeepL Pro

### 7.3 协作工具
- 使用 GitHub 进行文档协作
- 使用 Pull Request 进行文档审查
- 使用 Issues 进行问题跟踪

## 10. 持续改进

### 10.1 反馈收集

- 反馈渠道包括：
  - GitHub Issues

### 10.2 改进措施

## 11. 附录

### 11.1 文档模板
```markdown
# 文档标题

## 文档信息
- **版本**: 1.0.0
- **最后更新**: YYYY-MM-DD
- **作者**: 作者姓名
- **审核**: 审核人姓名

## 摘要
简短描述文档的主要内容和目的。

## 正文内容
文档的详细内容...

## 更新日志
| 日期 | 版本 | 更新内容 | 作者 |
|------|------|----------|------|
| YYYY-MM-DD | 1.0.0 | 初始版本 | 作者姓名 |
| YYYY-MM-DD | 1.1.0 | 新增内容 | 作者姓名 |
```

```markdown
| Liquidity | 流动性 | 资产的可交易性 |
```
