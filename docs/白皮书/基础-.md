直奔结论。下面是**这个项目目前“已经确定的 / 尚未确定的”边界清单**。这是可以直接作为你接下来设计、对外表达、砍需求的**判定基准**。

------

## 一、已经确定的（不可逆、可执行）

### 1️⃣ 项目的本质已经确定

**不是融资平台，不是单一预测市场，而是：**

> **“用可交易的阶段事件，做项目发现与信息聚合的系统”**

核心价值只有一个：
**把不确定的项目进展，拆成可验证、可交易、可传播的事件。**

------

### 2️⃣ 基础资产结构已确定

**事件 = 最小原子单位**，而不是“项目成功概率”。

每个事件必须具备三要素（缺一不可）：

- 截止时间（deadline）
- 证据标准（evidence spec）
- 裁决机制（resolver / attestor）

👉 不满足这三条的，**不能上链、不能交易、不能收钱**。

------

### 3️⃣ 资产形式已确定

- **YES / NO 成对资产**（可用 NFT / SFT / 半同质）
- 有到期、有结算、有归零
- 不是无限增发的代币

这点已经锁死，回不到“普通 token + AMM”。

------

### 4️⃣ 冷启动路径已确定

**三阶段演化是正确方向：**

1. **阶段 1：系统点火**
   - 系统是对手盘
   - 目标：让概率能动、让信息浮现
2. **阶段 2：人对人**
   - 用户互为对手盘
   - 系统只做规则与裁决
3. **阶段 3：系统退出**
   - 市场自洽
   - 项目留下的是「历史概率 + 交易记录 + 证据」

这是你这个项目能否规模化的生死线，已经确定。

------

### 5️⃣ 理论内核已确定

你选用的是：

- **Market Scoring Rule**
- 且只能是 **Logarithmic（对数）这一支**

原因不是“学术好看”，而是工程必需：

- 事件可拆分、可组合
- 条件事件不会污染无关事件
- 多维事件不会指数级烧钱

这在你当前设计里，已经是**唯一可用解**。

------

## 二、尚未确定的（需要继续设计 / 试错）

### 1️⃣ 事件粒度（最关键的未决项）

还没完全确定的是：

- 一个项目拆多少个事件才“刚好”
- 事件太少 → 信号粗糙
- 事件太多 → 用户疲劳 + 复杂度爆炸

👉 这是**产品而非数学问题**，需要真实项目跑。

------

### 2️⃣ 单事件的定价参数

包括：

- 单张票面金额（$1 / $10 / 动态）
- b 参数（系统最大风险敞口）
- 单事件最大系统亏损上限

这些**不是理论能给出最优解的**，只能通过真实交易校准。

------

### 3️⃣ NFT vs 半同质 vs 账本式

方向是明确的，但具体落地还在摇摆：

- 完全 NFT：可视化强，流动性弱
- 半同质 / 内部账本：交易顺滑，但链味弱

👉 这是**实现层面的分叉**，不是方向分叉。

------

### 4️⃣ 裁决权的归属

还未完全钉死的是：

- 纯系统裁决
- 多签 / 仲裁人
- 社区仲裁 + 押金

这是**法律 / 信任 / 成本**三者的博弈，目前还需要保守推进。

------

### 5️⃣ 用户为什么“持续来玩”

已经知道他们能赚钱，但还没完全确定：

- 他们是为了套利？
- 为了信息优势？
- 为了参与项目早期？

👉 激励叙事还在收敛中，但方向是对的。

------

## 三、当前阶段一句话判断

> **结构已经对了，但参数还不能定死。**

你现在不缺方向、不缺理论、不缺结构。
你缺的是：**用最小实现，把“不确定项”一条条跑成确定项。**

下一步不该再加新概念了，而是：

- 固定一套事件模板
- 固定一套参数区间
- 上真实项目，允许犯小错

你已经站在正确的轨道上，只差让车动起来。
