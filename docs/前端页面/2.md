# 项目首页（Project Page）UI 信息架构图 v0.1

## 0) 页面目标（写给前端的）

用户打开一个项目页，3 秒内要看懂：

1. 这个项目现在“哪几件事在发生”（事件矩阵）
2. 哪些事件“最有波动/最能赚”（机会提示）
3. 我该做什么（买/卖/挂单/提交证据/围观窗口）

# 2) 核心组件拆解（前端组件树）

### `ProjectPage`

* `ProtocolFooter`

# 3) EventCard（事件卡片）必须包含什么

> 事件卡片就是你的“发动机仪表盘”。每张卡让用户立刻知道：现在价格、证据、动作入口。

### EventCard 结构（建议固定顺序）

1. **Title 行**

* `topic_icon + event_name`
* `phase_badge`：P1/P2/P3
* `status_badge`：OPEN / ENDING / FROZEN / RESOLVED

2. **Price 模块（最重要）**

* `YES_price` / `NO_price`（或 bucket 分布）
* `24h_change`（涨跌幅）
* `spread / depth`（薄厚提示：Thin/OK/Deep）

3. **Evidence 进度**

* `evidence_progress`：已提交/需提交
* `last_evidence_time`
* `submit_evidence CTA`（按钮必须显眼）

4. **Deadline**

* `deadline_countdown`
* `dispute_window`（如果已到期）

5. **Action 区**

* `Buy YES` `Buy NO`
* `Sell`（如果用户持仓）
* `Place Order`（挂单入口）
* `View Orderbook`（次入口）

6. **Mini Meta（可信度/裁决）**

* `resolver/attestors`（头像/数量）
* `evidence_spec`（点击展开 checklist）

---

# 4) OpportunityStrip（赚钱点提示）怎么做才“真有效”

这条不是装饰，是拉动留存的钩子。建议固定 4 个槽位：

1. `Highest Volatility`：今日波动最大事件
2. `Thin Orderbook`：挂单薄的那一侧（适合做市吃差价）
3. `New Evidence`：最新证据更新（信息差入口）
4. `Index Mispricing`：Index 与底层偏离（套利入口，可选）

每个槽位点进去，直接滚动定位到对应 EventCard，并高亮 3 秒。

---

# 5) WindowControlPanel（窗口聚焦入口）

### 展示字段

* `next_window_start/end`
* `phase`（P1/P2/P3）
* `system_role`（FULL / BACKSTOP / OFF）
* `today_windows`（2 个点阵即可）
* CTA：

  * 窗口内：`Enter Window`（跳到“窗口交易视图”或开启高亮模式）
  * 窗口外：`Set Reminder`（可选，不做也行）

> 窗口内 UI 建议加“直播态”效果：交易流更靠前、EventCard 边框高亮、机会提示刷新频率更高。

---

# 6) EvidenceFeed（证据流）怎么设计成“可参与”

证据流每条要像 GitHub PR 一样，让人想点：

* `type`（URL/HASH/ONCHAIN/ATTESTATION）
* `submitted_by`
* `time`
* `targets`（对应事件）
* `status`（Pending/Accepted/Rejected）
* `impact_hint`（这条证据可能推动 YES/NO 哪边，*可以是社区投票，不必系统判断*）

并提供两个动作：

* `Support`（给证据点赞/质押支持，做成轻量信号）
* `Challenge`（发起质疑，进入 dispute 视图）

---

# 7) Index/Basket Panel（上层封装）最小可用长什么样

* `Index Price`
* `Constituent list`（事件 + outcome + 权重）
* `Mint` / `Redeem`（按钮）
* `Deviation hint`：Index - Σ(legs) 的偏离（有就显示，没有就不显示）

---

# 8) 路由建议（前端开撸最顺）

* `/p/:projectId` 项目首页（你现在要做的）
* `/p/:projectId/e/:eventId` 事件详情页（Orderbook + Evidence checklist + Resolver）
* `/p/:projectId/window` 窗口模式页（可选，用同组件但信息更集中）
* `/portfolio` 我的持仓与挂单（必须有，不然用户不会频繁回来）

---

# 9) 页面需要的最小数据接口（让后端知道怎么喂）

### `GET /api/project/:id`

* project 基本信息 + links + tags
* `phase_summary`（项目级别可选）

### `GET /api/project/:id/events`

每个 event 至少给：

* `event_id, topic, claim, type`
* `status, phase, system_role`
* `price_yes, price_no`（或 buckets）
* `change_24h`
* `deadline`
* `orderbook_thickness`（thin/ok/deep）
* `evidence_progress, last_evidence_at`
* `resolver_summary`

### `GET /api/project/:id/evidence?limit=…`

* evidence items list

### `GET /api/project/:id/activity?limit=…`

* trades/orders feed

### `GET /api/project/:id/index`

* index price + constituents + deviation
